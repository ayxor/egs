# Video Editor Service

Video Editor is the media-processing branch. It accepts a source file, applies operations such as watermarking, and writes the result to a destination URL.

Composer is the main caller. The service is business-agnostic and authenticated with an API key.

The current code exposes a small HTTP API plus a status/metrics surface for orchestration and health checks.

## API Contract

The contract is documented in this README and in the service code; keep any generated OpenAPI file beside the branch if you add one later.

Internal base URL: `http://video-editor:8080`

Required header for all endpoints:

```http
X-API-Key: <key>
```

## Endpoints

| Method | Path | Description |
|---|---|---|
| POST | /jobs | Create a job with an ID generated by the service |
| GET | /jobs/{job_id} | Get job state, progress, and requested operations |
| GET | /jobs/operations | Operations supported by this instance |

### Behavior Notes

- `POST /jobs` accepts multipart uploads and creates a new asynchronous job.
- `GET /jobs/<job_id>/progress` streams progress updates for Composer to relay.
- `GET /jobs/<job_id>/result` returns the processed media bytes.
- `GET /jobs/<job_id>/thumbnail` returns the generated thumbnail.
- `GET /metrics` is the Prometheus scrape target.

## Request Models

### POST /jobs (multipart/form-data)

Required form fields:

- `file` (binary) — source video file
- `operations` (JSON string) — non-empty array of operations to apply

Example:

```bash
curl -X POST "http://localhost:8080/jobs" \
  -H "X-API-Key: stub-api-key" \
  -F "file=@./sample.mp4" \
  -F 'operations=[{"type":"watermark","params":{"text":"UA","position":"bottom-right","opacity":0.8}}]'
```



## Job State

Response example for GET /jobs/{job_id}:

```json
{
  "job_id": "job_7f3a1b",
  "status": "processing",
  "percent": 45,
  "message": "Encoding video streams...",
  "operations": [
    {
      "type": "watermark",
      "params": {
        "text": "UA",
        "position": "bottom-right",
        "opacity": 0.8
      }
    }
  ],
  "created_at": "2026-03-17T10:00:00Z",
  "updated_at": "2026-03-17T10:01:23Z"
}
```

Possible states:

- queued
- processing
- done
- failed

## Expected HTTP Codes

| HTTP Code | Meaning | Typical cases in this API |
|---|---|---|
| 200 OK | Successful read | `GET /jobs/{job_id}`, `GET /jobs/operations` |
| 202 Accepted | Accepted for async processing | `POST /jobs` |
| 400 Bad Request | Invalid request payload | Missing `file` or `operations` in `POST /jobs` |
| 401 Unauthorized | Missing or invalid API key | Any endpoint without correct `X-API-Key` |
| 404 Not Found | Resource does not exist | Unknown `job_id` |
| 500 Internal Server Error | Server-side failure | Processing failed or OS resource issues |

## Design Notes

- `POST /jobs` receives the source file directly and always generates a `job_id`.
- `GET /jobs/{job_id}` includes both `operations` and progress state.
- There is no `/resources/{resource_id}/state` endpoint.
- The service manages processing jobs but not file ownership.

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python3 app.py
```

```bash
export VIDEO_EDITOR_API_KEY="my-key"

```



### Basic Frontend Demo

This service includes a small frontend page served by Flask at:

```text
http://localhost:8080/
```

The page is meant only as a quick proof that the API works. It lets you:

- set `X-API-Key`
- load supported operations (`GET /jobs/operations`)
- create a job (`POST /jobs`, multipart upload)
- watch progress (`GET /jobs/{job_id}` polling)

Use the same API key configured in the backend (`VIDEO_EDITOR_API_KEY`, default `stub-api-key`).

## Notes

- This service is run from the `video-editor` branch through the `main` stack compose file.
- It does not own storage; it only processes bytes and returns results.
- In the public stack, requests still flow through Traefik at `uastream.com` before reaching the orchestrated services.

## Diagrams

### Service Architecture & Flowchart

The Video Editor exposes stateless HTTP endpoints. On `POST /jobs`, it saves the source file to an on-disk job directory under `/tmp/video_editor_jobs/{job_id}/`, registers the job in an in-memory store, and spawns a background thread that runs the FFmpeg pipeline. Progress is streamed to the caller over a Server-Sent Events (SSE) endpoint. Once processing completes, the result video and thumbnail are passively served from disk until the Composer pulls them.

```
Composer
  │
  ├─ POST /jobs (multipart: file + operations)
  │     └─> Save src.mp4 to disk, register job in memory, spawn FFmpeg thread
  │
  ├─ GET /jobs/{id}/progress   (SSE stream)
  │     └─> Tail FFmpeg stderr, emit {status, percent, message} events
  │
  ├─ GET /jobs/{id}/result     (on status=done)
  │     └─> Stream dst.mp4 bytes
  │
  └─ GET /jobs/{id}/thumbnail  (on status=done)
        └─> Stream thumb.jpg bytes
```
